# Plan: Build Shane's Lane & Full Pipeline

## Goal
Build Shane's personalized music lane from existing Spotify + Last.fm connections, then wire the full daily pipeline.

## Current State
- **Providers:** Spotify (working, 50 tracks via search), Last.fm (working, 50 recent tracks)
- **DB:** Aiven PostgreSQL 17.10 with tables: `tracks`, `listens`, `feedback_events`, `lanes`
- **Existing modules:** ingestion (with dedup), scoring (FR-7), feedback (decay), scheduler DAG
- **Missing:** session builder, lane extraction, actual pipeline execution, provider registry

## Approach
1. **Ingest real data** — run ingestion from both providers into PostgreSQL
2. **Build sessions** — group listens by 30-min gap rule
3. **Extract lanes** — cluster sessions into Shane's listening lanes
4. **Wire pipeline** — connect all DAG steps into a runnable daily job
5. **Verify** — test end-to-end with real data

## Step-by-Step

### Step 1: Provider Registry + Ingestion Runner
**File:** `backend/adapters/__init__.py`
- Create a registry that maps provider names to adapter instances
- Auto-discover enabled providers from config
- Provide `fetch_all_listens(user_id)` that aggregates from all providers

**File:** `backend/ingestion.py` (extend)
- Add `run_ingestion(user_id)` that:
  1. Gets all enabled adapters from registry
  2. Calls each adapter's `fetch_listens()`
  3. Passes to existing `ingest_listening_history()`
  4. Returns summary stats

**File:** `backend/db/session.py` (extend)
- Add `get_engine()` helper to reuse DB engine
- Add `get_session()` for convenience

### Step 2: Session Builder
**File:** `backend/sessions/__init__.py` (new)
- `build_sessions(user_id, db)` → list of sessions
- Each session: tracks, timestamps, duration, skips, replay behavior
- Rule: gap > 30 min = new session boundary
- Store in new `sessions` table (or extend `listens` with session_id)

### Step 3: Lane Extraction Engine
**File:** `backend/lane_extraction/__init__.py` (new)
- `extract_lanes(user_id, db)` → list of lanes
- Uses session data + track metadata
- Clustering approach:
  - Phase 1: Simple rule-based lane extraction (time-based, genre-based)
  - Phase 2: ML-based (UMAP + HDBSCAN) when data volume is sufficient
- Lane properties: name, description, top artists, tags, energy profile, confidence
- Store in `lanes` table

### Step 4: Full Pipeline Runner
**File:** `backend/pipeline/__init__.py` (new)
- `run_daily_pipeline(user_id)` → execution of full DAG
- Each step calls the appropriate module
- Returns detailed results per step
- Idempotent (can be re-run safely)

### Step 5: API Endpoint for Pipeline
**File:** `backend/api/routes/pipeline.py` (new)
- `POST /pipeline/run` — run daily pipeline for a user
- `GET /pipeline/status` — check last run status
- `GET /pipeline/lane` — get Shane's current lane

### Step 6: Tests
- Test ingestion with real provider data
- Test session building with known data
- Test lane extraction with synthetic data
- Test full pipeline end-to-end

## Risks & Tradeoffs
- **Data volume:** Lane extraction needs enough data. Start with rule-based, upgrade to ML later
- **Spotify OAuth:** Current Spotify adapter uses public search. OAuth2 user auth needed for real history
- **Last.fm rate limits:** 50 tracks per call, may need pagination
- **Session table:** Could add `session_id` to `listens` table instead of new table (simpler)
